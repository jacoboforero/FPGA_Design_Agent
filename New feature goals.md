# New Feature Goals

This document expands the feature backlog discussed on April 19, 2026 into concrete gaps between the current repo and stronger agentic RTL-generation systems. Each goal is framed with:

- what the repo does today
- what stronger systems or current best practices do differently
- what the goal would look like in this architecture

## Adaptive Selective Spend

- Current repo example: the repo already supports per-role model routing through `config/domains/agents.yaml`, `core/runtime/config.py`, and `agents/common/llm_gateway.py`, but the checked-in config leaves the role-level `model` fields as `null`, so the effective behavior is one default model for most LLM-backed stages. Worker gateways are also initialized once per role, which means model choice is effectively static for the life of the worker.
- Counterexample from stronger systems: escalation-centric systems such as HDLFORGE do not just assign one model per role. They spend cheaply on easy cases, then escalate only when diagnostics, repeated failures, or difficult subproblems justify the extra cost. Major-lab guidance around coding agents also pushes in this direction: spend more only when the current context indicates it is necessary.
- Goal in this repo: add a runtime policy layer above the current role-based routing so implementation, reflection, and debug can change model/provider based on retry count, mismatch severity, repeated failure signatures, stuck detection, benchmark mode, or other hard-case triggers. This can preserve the current orchestrator and queue architecture while making model spend responsive to run state rather than fixed by YAML alone.

## Add Selective Candidate Generation Under Tight Gating

- Current repo example: benchmark mode already supports higher-variance sampled settings, but the core generation architecture is still primarily single-candidate generation followed by repair. The repo does not yet have a built-in policy that says "only generate alternatives when the first attempt fails, looks uncertain, or enters a hard-case bucket."
- Repo counterexample that shows the risk: broad candidate generation can turn into a token-spend benchmark trick if it is applied blindly and then reranked by the same downstream oracle. That may raise scores, but it does not necessarily improve the base prompting, planning, or repair quality of the system.
- Counterexample from stronger systems: search-based systems such as EvolVE use candidate generation as part of a larger verification-and-refinement strategy, not just as brute-force repeated prompting. The better pattern for product systems is selective and bounded search, where alternative candidates are generated only under explicit escalation conditions and are screened with independent validation signals.
- Goal in this repo: add a tightly gated candidate-generation policy rather than a blanket `n`-samples strategy. Good triggers would be repeated failures, uncertainty heuristics, severe mismatch clusters, or planner/debug signals that a case is unusually hard. Candidate pools should stay small, costs should be explicit, and reranking should rely on independent checks such as lint, interface fidelity, formal properties when available, or partial validation signals instead of simply brute-forcing more tokens on every run.

## Improve Context Editing

- Current repo example: the debug worker reads large artifacts and then truncates them to fixed lengths before prompting, for example `rtl_prompt`, `tb_prompt`, `lint_log_prompt`, `sim_log_prompt`, `distilled_prompt`, and `reflection_prompt` in `agents/debug/worker.py`. The reflection worker similarly caps RTL and testbench excerpts with `_number_lines_limited(..., max_lines=450)` in `agents/reflection/worker.py`. This is context control, but it is mostly truncation plus concatenation.
- Repo counterexample that shows the ceiling: the distill worker already extracts focused evidence such as failure cycle, failure time, log excerpt, waveform excerpt, and signal snapshots in `workers/distill/worker.py`, but the debug path still sends large raw artifacts alongside that distilled view instead of aggressively replacing raw context with the smallest relevant slices.
- Counterexample from stronger systems: better agentic systems and current context-management guidance do more than clip inputs. They rewrite context around the active failure, keep only the relevant code regions, and localize evidence to the exact failing signals, states, or temporal window. Systems like Veri-Sure explicitly emphasize localized dependency slicing and temporal tracing rather than broad raw-context dumps.
- Goal in this repo: move from coarse truncation to targeted context editing. For example, instead of sending the first 18k characters of RTL and TB, send only the always block or combinational region driving the failing signal, only the exact waveform window around the violation, only the most relevant testbench checker region, and only the specific prior patch summary tied to the current failure signature. This should reduce token usage and improve repair precision without changing the overall repair loop.

## Add Structured Working Memory

- Current repo example: the orchestrator keeps useful but thin retry state in `attempt_history_by_node`, including `failure_signature`, `failure_line`, `touched_files`, `patch_summary`, stage outcomes, and `stuck` metadata in `orchestrator/orchestrator_service.py`. This is a good start, but it is not yet a full working memory for the node.
- Repo counterexample that shows the gap: the debug and reflection agents still reconstruct the current situation from raw logs, code, and attempt history each time. They do not inherit a structured memory object that says, for example, "reset semantics are now believed correct," "the previous patch changed only TB timing," "the counter rollover hypothesis was falsified," or "the likely root cause has narrowed to output decode logic."
- Counterexample from stronger systems: stronger multi-agent and contract-aware systems maintain more explicit intermediate state. HDLFORGE describes contract and memory management; contract-aware systems like Veri-Sure maintain localized reasoning artifacts rather than forcing every stage to rediscover the problem from scratch.
- Goal in this repo: add a per-node working memory artifact that survives across distill, reflection, debug, and retry. It should hold current hypotheses, ruled-out theories, inferred timing/reset semantics, failing outputs, likely dependency slice, what changed on the last patch, and what still has not changed despite retries. That would make the repair loop cumulative instead of mostly reconstructive.

## Add Durable Failure Memory

- Current repo example: the repo already has durable memory for passing designs through the finalizer and RAG archive path in `agents/finalizer/worker.py` and `adapters/rag/rag_service.py`. This helps future generation stages retrieve prior good designs and verification patterns.
- Repo counterexample that shows the gap: failure knowledge is mostly ephemeral. The orchestrator tracks recent failure signatures in-memory for the current node, but the repo does not persist recurring failure classes and successful fix strategies as a retrieval target. Once a run ends, most of that debugging intelligence is not turned into reusable memory.
- Counterexample from stronger systems: stronger repair-oriented systems do not only remember successful outputs; they also remember how failures presented and what fixed them. This is especially important in domains with recurring bug families such as reset handling, FSM transitions, width/sign mismatches, TB race conditions, and interface/header drift.
- Goal in this repo: build a compact failure-memory store alongside the current passing-design archive. Each entry could include failure signature, affected signals, distilled failure summary, patch summary, changed files, and whether the fix succeeded. Retrieval could then surface not just "similar passing RTL" but also "similar failure pattern and repair strategy."

## Add Formal Verification

- Current repo example: today verification is based on linting, semantic lint, testbench generation or oracle TB binding, simulation via `iverilog`/`vvp`, and acceptance gating. The planner/spec flow carries an `assertion_plan` field, but there is no active formal engine in the runtime path that proves properties or checks equivalence against those assertions.
- Repo counterexample that shows the gap: if simulation passes for the current testbench, the repo accepts that as the primary correctness signal. That is practical, but it means hidden behaviors outside the test stimulus can survive, especially on tricky temporal edge cases.
- Counterexample from stronger systems: FormalRTL, Veri-Sure, and other newer RTL-generation systems use formal methods as a major differentiator. Formal verification lets them prove or refute properties beyond the observed tests and often gives much sharper signals than simulation alone.
- Goal in this repo: add a formal-verification layer that can consume planned assertions/contracts and check safety, bounded behavior, or equivalence-style properties. This should sit next to the current lint and simulation path rather than replacing it. Even a limited first step would be valuable if it turns existing spec/planner assertions into machine-checked properties instead of unused planning metadata.

## Add Counterexample-Guided Repair

- Current repo example: when simulation fails, the repo distills logs and waveforms in `workers/distill/worker.py`, reflects on them in `agents/reflection/worker.py`, and patches code in `agents/debug/worker.py`. This is best described as simulation- and waveform-guided repair.
- Repo counterexample that shows the gap: the debug loop can tell that a signal mismatched or that a timeout occurred, and it can inspect local evidence, but it does not receive a formal counterexample trace that proves exactly how a property is violated under a precise sequence of inputs and states.
- Counterexample from stronger systems: systems that combine formal checking with repair use the generated counterexample as the main repair artifact. That is a stronger feedback signal than a broad simulation mismatch because the violating trace is already minimized around a specific broken property.
- Goal in this repo: once formal checking exists, feed formal counterexamples into the current distill/reflect/debug loop as first-class artifacts. This would let the repair loop target explicit violating traces rather than relying only on empirical testbench failures and waveform slices.

## Add Failure-Class-Specialized Repair Prompts

- Current repo example: the debug loop is driven by one large generic prompt in `agents/debug/worker.py`. It already contains useful general repair guidance and a few special-case rules such as `TBSEM008`, but most failure handling still flows through one broad repair prompt regardless of whether the problem is a timer, FSM, reset bug, combinational decode issue, or interface/header mismatch.
- Repo counterexample that shows the gap: the orchestrator already records failure signatures and stagnation state, but those signals are not yet used to switch the debug agent into specialized repair modes. The system knows something is repeating, but it does not yet know how to respond differently for distinct bug families.
- Counterexample from stronger systems: stronger repair-oriented systems often narrow the repair context and strategy by bug class. A counter/timer repair prompt should reason differently from an FSM repair prompt, and both should differ from interface/header normalization or testbench race repair.
- Goal in this repo: add targeted repair modes and prompt families for recurring bug classes such as counters/timers, FSMs, reset semantics, combinational decode, and interface/header repair. The current generic prompt can remain as a fallback, but failure signatures, lint outcomes, and distilled evidence should be able to route the debug stage into a more specialized repair instruction set when the bug class is recognizable.

## Unify Debug And Reflection Under The Prompt Registry

- Current repo example: implementation, testbench, spec-helper, and finalizer already use the prompt registry under `core/prompting` and `prompts/`, while debug and reflection still assemble prompts inline in `agents/debug/worker.py` and `agents/reflection/worker.py`.
- Repo counterexample that shows the gap: debug already has prompt fragments on disk under `prompts/fragments/debug/`, which suggests the migration is partially started but not fully landed. Reflection has no equivalent registry-backed prompt family yet, so its instructions are still embedded directly in worker code.
- Counterexample from stronger systems and best practices: stronger agentic systems benefit from versioned prompt assets, explicit schemas, and prompt traces that make iteration measurable. This is especially important in repair loops, where prompt changes can materially affect benchmark behavior and should be auditable.
- Goal in this repo: move debug and reflection prompts into the same registry-backed prompt infrastructure used by the other prompt families, with explicit schemas, versioned prompt assets, and traceable prompt metadata. Important distinction: this goal is about unifying prompt management, not merging the runtime responsibilities. Debug and reflection should remain two separate agents and two separate orchestrated tasks if that task split continues to make sense architecturally; only the prompt-definition and traceability layer should become unified.

## Make The Planner Agent More Sophisticated

- Current repo example: the planner is deterministic today, as documented in `agents/planner/worker.py`. That gives the repo stable artifact shapes and predictable planning behavior, which is valuable for reliability.
- Repo counterexample that shows the ceiling: deterministic planning works well when the spec is already structured enough, but it is weaker when decomposition, interface boundaries, or module roles are ambiguous. In those cases the repo depends more heavily on pre-structured specs and less on planner intelligence.
- Counterexample from stronger systems: systems like VerilogCoder push more intelligence into planning and decomposition rather than treating planning as a rigid translation layer. That matters most when the task requires architectural judgment rather than straightforward code emission.
- Goal in this repo: move toward a hybrid planner. Keep deterministic validation and schema output, but optionally add an LLM-assisted planning path for ambiguous, underspecified, or multi-module cases. The likely use cases are better module decomposition, stronger interface inference, richer implementation sketches, and clearer child-module contracts. This should remain constrained by the current artifact schemas so the rest of the runtime stays stable.

## Improve The Implementation Agent

- Current repo example: the implementation worker already uses the prompt registry and stage-specific RAG in `agents/implementation/worker.py`, and it applies useful fragments such as language rules and integration rules. That is a good baseline and is already more structured than many prototype systems.
- Repo counterexample that shows the gap: the actual implementation prompt is still very thin compared with the testbench prompt. `prompts/implementation/generate/system.md` is essentially a short role statement plus fragments, and `user.md` is mostly a raw field dump ending with "Implement the described RTL." By contrast, `prompts/testbench/generate/system.md` defines a clearer priority order, output contract, and behavioral constraints. So the implementation path uses the new infrastructure, but the prompt itself is not yet as contract-driven as the stronger prompt families in the repo.
- Counterexample from stronger systems: stronger RTL-generation systems usually enforce more structure before code emission. VerilogCoder emphasizes planning and structured analysis. EvolVE pushes iterative refinement rather than single-shot generation. Contract-aware systems are more explicit about interface preservation, temporal behavior, and failure localization.
- Goal option 1: keep the current single-step generation shape but upgrade the implementation prompt to be more state-of-the-art in structure. That means an explicit output contract, exact interface preservation rules, stronger timing/reset rules, clearer RTL-discipline rules, prioritized correctness criteria, and a built-in self-checklist before returning code.
- Goal option 2: move to a two-step pipeline in which the system first emits a structured design sketch or implementation plan, then generates RTL from that plan. This would help lock state semantics, edge-case handling, and temporal behavior before code emission.
- Important architecture note: that first sketch stage does not necessarily need to live inside the implementation agent. A suitable alternative is to enrich planner output so the planner emits a more detailed implementation plan up front and the implementation agent focuses on translation to RTL. That may be the cleaner fit if planning depth should remain centralized in the planner layer.
